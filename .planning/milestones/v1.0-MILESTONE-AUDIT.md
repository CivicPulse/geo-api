## Cross-Phase Integration

### Integration Findings

**14/16 exports properly wired.** Two integration gaps found:

#### Gap 1: admin_overrides Table Never Written by API (HIGH)

**Affected:** DATA-03, GEO-07

The `AdminOverride` ORM model and `admin_overrides` table exist. The CLI import checks this table before setting official geocoding (`cli/__init__.py:186–189`). But `GeocodingService.set_official` (the GEO-07 path) stores admin overrides as `GeocodingResult` rows with `provider_name="admin_override"` — it never writes to `admin_overrides`.

**Impact:** The CLI's admin-override guard always returns NULL. Currently harmless (ON CONFLICT DO NOTHING preserves existing official), but the guard logic is silently broken.

**Fix:** Either populate `admin_overrides` in `GeocodingService.set_official`, or change the CLI to check for `provider_name="admin_override"` in `geocoding_results` instead.

#### Gap 2: Official Geocoding Priority — First Writer Wins (HIGH)

**Affected:** DATA-03

Both the CLI import and `GeocodingService` use `ON CONFLICT (address_id) DO NOTHING` for `official_geocoding`. This means the first source to set an official record wins permanently (unless admin overrides via PUT endpoint).

**Impact:** If an address is geocoded via the API before GIS import, census remains official. DATA-03 requires county GIS to be the default when no admin override exists — this only works if GIS is imported first.

**Fix:** CLI import should use `ON CONFLICT (address_id) DO UPDATE SET geocoding_result_id = :id` when the existing official is not an admin override.

  Create the single `prod` overlay (no dev overlay — the civpulse-gis stack
  is shared, per locked decision D-03).

  1. **`k8s/osm/overlays/prod/kustomization.yaml`** — Minimal overlay that
     references the base:
     ```yaml

     kind: Kustomization

     resources:
       - ../../base
     ```
     The overlay exists mainly so ArgoCD has a stable path to target; no
     environment-specific patches needed since civpulse-gis is a single
     shared namespace.

  2. **`k8s/osm/overlays/prod/argocd-app.yaml`** — New ArgoCD Application
     based on `k8s/overlays/dev/argocd-app.yaml` pattern, with these
     specific values (per locked decisions D-03, D-04):
     ```yaml
     apiVersion: argoproj.io/v1alpha1
     kind: Application
     metadata:
       name: osm-stack
       namespace: argocd
     spec:
       project: default
       source:
         repoURL: https://github.com/CivicPulse/geo-api.git
         targetRevision: main
         path: k8s/osm/overlays/prod
       destination:
         server: https://kubernetes.default.svc
         namespace: civpulse-gis
       syncPolicy:
         automated:
           prune: true
           selfHeal: true
         syncOptions:
           - CreateNamespace=true
           - ServerSideApply=true
     ```

     Note: `CreateNamespace=true` + committed `namespace.yaml` is belt-and-
     suspenders (per D-02) — ArgoCD will apply the committed Namespace
     resource and the sync option is a safety net for fresh-cluster bootstrap.

     `ServerSideApply=true` per D-04 (best practice for modern ArgoCD).

  **IMPORTANT:** The overlay `kustomization.yaml` references `../../base`
  but the `argocd-app.yaml` is NOT a kustomize resource — ArgoCD
  Applications are applied directly, not built via kustomize. Do NOT list
  `argocd-app.yaml` in the overlay's `resources:`. Verify with kustomize
  build below that argocd-app.yaml does NOT appear in the built output.
